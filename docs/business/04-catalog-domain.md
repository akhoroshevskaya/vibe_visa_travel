# 1. Purpose

Catalog Domain управляет коммерческой моделью компании.

Он определяет:

- какие продукты предлагает компания;
- из каких услуг состоит каждый продукт;
- какие услуги могут продаваться отдельно;
- какой Workflow используется для выполнения продукта;
- какие коммерческие предложения доступны клиентам.

Catalog Domain не отвечает за выполнение услуг.

Он определяет **что продает компания**, а не **как выполняется работа**.

---

# 2. Business Questions

На какие вопросы отвечает домен?

- Какие продукты предлагает компания?
- Какие услуги входят в каждый продукт?
- Какие услуги можно приобрести отдельно?
- Какие услуги являются обязательными?
- Какие услуги являются дополнительными?
- Какой Workflow используется для данного продукта?
- Какие продукты доступны клиенту?
- Из чего формируется стоимость продукта?

# 3. Основные сущности

## Product

Product представляет коммерческий продукт компании.

Примеры:

- Шенгенская виза
- Виза Великобритании
- Виза США
- Виза Канады
- Виза Австралии
- Тур
- Тревел-консьерж
- Зарубежная банковская карта
- Карта АТЭС

---

## Service

Service представляет отдельную услугу.

Одна и та же услуга может использоваться в нескольких Product.

Примеры:

- визовая поддержка;
- запись на подачу;
- страховка;
- бронирование авиабилетов;
- бронирование отелей;
- перевод документов;
- нотариальные услуги;
- услуги курьера;
- доставка паспорта;
- оформление маршрута путешествия.

---

## Workflow Template

Workflow Template определяет бизнес-процесс выполнения Product.

Каждый Product использует один Workflow Template.

Один Workflow Template может использоваться несколькими Product.

Workflow Template определяет:

- последовательность Stage;
- обязательные этапы;
- условные этапы;
- логику прохождения Case.

---

# 4. Business Rules

### BR-001

Каждая Service может продаваться как самостоятельная услуга.

### BR-002

Product состоит из одной или нескольких Service.

### BR-003

Одна Service может использоваться в составе нескольких Product.

### BR-004

Состав Service внутри Product может отличаться в зависимости от выбранной конфигурации продукта.

### BR-005

Каждый Product использует один Workflow Template.

### BR-006

Один Workflow Template может использоваться несколькими Product.

### BR-007

Workflow Template не изменяет Workflow уже существующих Case.

# 9. Связи между сущностями

```text
Product
      │
      ├────────► Service
      │
      └────────► Workflow Template
                      │
                      ▼
                  Workflow Stages
```

Product определяет:

- что продается;
- какие Service входят;
- какой Workflow используется.

Operations Domain использует Workflow Template для создания Workflow конкретного Case.

### Дополнительные бизнес-правила

- Product определяет состав услуг, необходимых для выполнения заказа.
- При создании Case на основании выбранного Product автоматически создается соответствующий набор Case Service.
- Состав создаваемых Case Service определяется конфигурацией Product.

---

# 10. Архитектурные принципы

- Product является коммерческой единицей продажи.
- Service является переиспользуемой бизнес-услугой.
- Workflow Template определяет процесс выполнения Product.
- Catalog Domain не хранит информацию о клиентах или выполнении услуг.
- Все коммерческие предложения компании описываются только в Catalog Domain.
