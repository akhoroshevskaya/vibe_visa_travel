# AI Agent Guide

Version: 1.0

Project: V!BE Business OS

---

# Purpose

Этот документ определяет единый стандарт взаимодействия с AI-агентами в проекте V!BE Business OS.

Его цель — обеспечить:

- предсказуемую работу ИИ;
- единый стиль запросов;
- безопасность проекта;
- сохранение архитектурной целостности;
- возможность использовать разных AI-агентов (ChatGPT, Codex, Claude и др.) без изменения процессов.

---

# Основные принципы

## 1. Business First

Любое решение должно соответствовать утвержденной Business Architecture.

Архитектура имеет приоритет над кодом.

---

## 2. Source of Truth

Единственным источником истины являются документы проекта.

Если код противоречит документации — сначала анализируем причину, а не исправляем автоматически.

---

## 3. Минимум предположений

Если информации недостаточно:

- не придумывать;
- явно указать, чего не хватает;
- предложить варианты.

---

## 4. Безопасность

AI не должен выполнять потенциально опасные действия без явного запроса пользователя.

Например:

- удаление файлов;
- массовое переименование;
- изменение структуры проекта;
- изменение базы данных;
- git push;
- merge;
- удаление веток.

---

# Режимы работы

## Режим 1. Анализ

Используется по умолчанию.

AI может:

- читать файлы;
- анализировать проект;
- искать зависимости;
- составлять отчеты;
- искать несоответствия.

AI НЕ должен:

- изменять файлы;
- запускать команды;
- создавать коммиты.

Стандартная фраза:

> Не изменяй никаких файлов.

---

## Режим 2. Проектирование

AI может:

- предлагать архитектуру;
- проектировать модели данных;
- описывать структуры;
- готовить документы.

Изменения в проекте выполняются только после утверждения.

---

## Режим 3. Реализация

После утверждения AI может:

- создавать новые файлы;
- редактировать существующие;
- писать код;
- обновлять документацию.

---

## Режим 4. Рефакторинг

Допускается только после явного подтверждения.

AI обязан объяснить:

- зачем нужен рефакторинг;
- что изменится;
- какие риски существуют.

---

# Стандарт постановки задач

Каждая задача должна содержать четыре части.

## Цель

Что необходимо получить.

## Контекст

Какие документы необходимо учитывать.

## Ограничения

Что запрещено делать.

## Ожидаемый результат

Что должно получиться.

---

Пример:

Цель:

Проанализировать Business Information Model.

Контекст:

Использовать документы из docs/business.

Ограничения:

Не изменять файлы.

Ожидаемый результат:

Список бизнес-сущностей и их связей.

По умолчанию запрещено:

- commit;
- push;
- merge;
- rebase;
- delete.

Если требуется изменение Git — пользователь должен указать это явно.

режим Read Only.

AI может:

- анализировать структуру;
- искать ошибки;
- сравнивать модели.

AI не должен создавать или изменять базы данных без подтверждения.

# Работа с документацией

- обновить README;
- проверить актуальность архитектурных документов;
- сформировать Status Report.

# AI Working Agreement

Перед началом любой новой задачи AI должен определить режим работы.

В проекте используются четыре режима.

## Mode 1 — Analyze

Используется по умолчанию.

Разрешено:

- читать файлы;
- анализировать проект;
- искать зависимости;
- формировать отчеты.

Запрещено:

- изменять файлы;
- запускать команды;
- выполнять git-операции.

---

## Mode 2 — Design

Используется при проектировании.

Разрешено:

- проектировать структуры;
- предлагать архитектурные решения;
- создавать черновики документов.

Все изменения требуют утверждения пользователя.

---

## Mode 3 — Implement

Используется после утверждения решения.

Разрешено:

- создавать новые файлы;
- изменять существующие;
- писать код;
- обновлять документацию.

После завершения AI обязан перечислить все внесенные изменения.

---

## Mode 4 — Review

Используется для проверки качества.

AI должен:

- искать ошибки;
- искать противоречия;
- искать нарушения архитектуры;
- искать нарушения Business Rules.

Изменения не выполняются автоматически.

---

# Default Behaviour

Если пользователь явно не указал режим работы,
используется:

Mode 1 — Analyze.

AI никогда не должен переходить к изменению проекта самостоятельно.

---

# Repository Rules

Перед выполнением задачи AI обязан:

1. Изучить README проекта.
2. Проверить ADR при наличии архитектурных изменений.
3. Использовать Business Information Model как источник истины.
4. Не создавать новые сущности без необходимости.
5. Не перепроектировать утвержденную архитектуру.

Используется при проверке архитектуры.

AI должен:

- искать противоречия между документами;
- искать дублирование;
- искать устаревшие решения;
- искать несогласованность терминов;
- искать нарушения ADR.

AI НЕ должен:

- самостоятельно исправлять документы;
- менять архитектуру;
- делать вывод о наличии ошибки без ссылок.

Каждое замечание должно содержать:

1. Документ
2. Раздел
3. Цитату
4. Объяснение
5. Уровень серьезности

Mode: Architecture Review

Не изменяй никаких файлов.

Для каждого найденного несоответствия:

1. Укажи документ.
2. Укажи раздел.
3. Приведи цитату.
4. Объясни, почему считаешь это несоответствием.
5. Оцени критичность:
   - Critical
   - High
   - Medium
   - Low
6. Не предлагай исправлений.
