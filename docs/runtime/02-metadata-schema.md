# Metadata Schema

## Назначение

**Metadata Schema** определяет единый набор обязательных метаданных для архитектурной документации проекта **V!BE Business OS**.

Использование единой схемы обеспечивает согласованность документации, упрощает навигацию по архитектуре и создает основу для автоматической проверки документов.

---

## Цели

Схема метаданных используется для:

- стандартизации архитектурной документации;
- идентификации документов;
- отслеживания жизненного цикла документов;
- поддержки Architecture Review;
- подготовки архитектуры к автоматической обработке.

---

## Обязательные метаданные

Каждый архитектурный документ должен содержать следующий набор метаданных.

| Поле | Обязательное | Описание |
|------|:------------:|----------|
| Document ID | ✅ | Уникальный идентификатор документа |
| Title | ✅ | Название документа |
| Type | ✅ | Тип архитектурного документа |
| Layer | ✅ | Архитектурный слой |
| Status | ✅ | Текущий статус документа |
| Owner | ✅ | Ответственный за документ |
| Version | ✅ | Версия документа |
| Last Updated | ✅ | Дата последнего обновления |
| Next Review | ⬜ | Плановая дата следующего ревью |

---

## Типы документов

Для архитектурной документации используются следующие типы.

| Тип | Назначение |
|------|------------|
| Vision | Видение проекта |
| Architecture | Архитектурный документ |
| Runtime Specification | Документ Runtime |
| Repository Specification | Описание структуры репозитория |
| Standard | Архитектурный стандарт |
| Governance | Правила управления архитектурой |
| Engineering | Документ реализации |
| Decision Record | Архитектурное решение (ADR) |

---

## Архитектурные слои

Каждый документ относится к одному архитектурному слою.

- Business
- Business OS
- Runtime
- Engineering
- Implementation

---

## Статусы документов

Используются следующие статусы жизненного цикла.

| Статус | Значение |
|---------|----------|
| Draft | Документ создается |
| Review | Находится на архитектурном ревью |
| Approved | Утвержден |
| Deprecated | Больше не используется |

---

## Правила идентификации

Для каждого документа должны соблюдаться следующие правила:

- Document ID должен быть уникальным;
- название должно быть уникальным в пределах архитектурного слоя;
- версия изменяется при каждом утвержденном обновлении;
- статус отражает текущее состояние документа;
- дата последнего обновления должна поддерживаться в актуальном состоянии.

---

## Пример метаданных

```text
Document ID: RT-002
Title: Metadata Schema
Type: Runtime Specification
Layer: Runtime
Status: Approved
Owner: Enterprise Architecture
Version: 1.0
Last Updated: 2026-07-18
Next Review: Before MVP Development
```

---

## Использование схемы

Metadata Schema применяется:

- ко всем архитектурным документам;
- при проведении Architecture Review;
- при Validation;
- при обновлении документации;
- при сопровождении проекта.
